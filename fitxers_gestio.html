Gestió de fitxers en Java

Ara que ja sabem llegir i escriure de fitxers, potser comença a ser hora de descobrir com podem saber coses dels fitxers, com ara si existeixen, quina mida tenen i si poden ser llegits o escrits abans d'intentar-ho.

Hi és o no hi és

En els exercicis que hem fet fins ara que involucraven fitxers hem hagut de suposar que els fitxers existien, però sovint no podem suposar-ho i ens caldrà comprovar-ho.

Considera aquesta nova versió del programa Mostrat.java:

MostratSiEstas.java
 1/*
 2 * programa que mostra el seu propi codi
 3 * Nota: en cas que el codi font no es trobi a la carpeta actual, mostra un error
 4 */
 5import java.io.BufferedReader;
 6import java.io.File;
 7import java.io.FileReader;
 8import java.io.IOException;
 9public class MostratSiEstas {
10    public static void main(String[] args) throws IOException {
11        String cami = "MostratSiEstas.java";
12        File fitxer = new File(cami);
13        if (! fitxer.exists()) {
14            System.out.println("Error: no es troba el fitxer: " + cami);
15            return;
16        }
17        FileReader fileReader = new FileReader(fitxer);
18        BufferedReader input = new BufferedReader(fileReader);
19        while (true) {
20            String linia = input.readLine();
21            if (null == linia) break;
22            System.out.println(linia);
23        }
24        input.close();
25    }
26}

Observa com el codi fa servir la variable de tipus File. A la línia 12 la declara i inicialitza, especificant que el fitxer estarà associat al camí MostratSiEstas.java. A la línia 13 comprova si el fitxer existeix. Això ho fa tot cridant a la funció booleana exists() associada als fitxers (File`).

La crida fitxer.exists() retorna cert quan es troba el fitxer i fals altrament.

És important entendre que el camí no té perquè existir per tal de crear un File associat. Per exemple, considera la següent interacció:

jshell> File fitxer = new File("/un_fitxer_inexistent")
fitxer ==> /un_fitxer_inexistent

jshell> fitxer.exists()
$1 ==> false

Com pots veure, inicialitzar fitxer no ha suposat cap problema. En canvi, com era d'esperar, en comprovar la seva existència ens retorna que no hi és.

Encara més, el camí no necessàriament té perquè correspondre a un fitxer tal i com l'entenem. Per exemple, podria ser un directori:

jshell> File fitxer = new File("/tmp")
fitxer ==> /tmp

jshell> fitxer.exists()
$1 ==> true

Podem entendre un File com un camí o ruta (path) que permet especificar un element del sistema de fitxers que, potser ni hi és.

Finalment, com que en realitat no estem obrint el fitxer fins que no li passem a fileReader, no cal tancar-lo.

Per cert, t'has fixat que ara FileReader rep a la línia 17 un File en comptes d'un String? Ja que tens el File pots escollir si li passes el camí o el fitxer.

És un fitxer o un directori?

Ara que ja hem vist que File pot ser tant un fitxer com un directori, potser seria interessant poder-ho distingir. File ens ofereix un munt d'utilitats interessants a banda de exists().

Per exemple, fixa't en aquesta interacció

shell> File fitxer = new File("/tmp")
fitxer ==> /tmp

jshell> fitxer.isDirectory()
$1 ==> true

jshell> fitxer.isFile()
$2 ==> false

jshell> new File("/usr/bin/java").isFile()
$3 ==> true

Fixa't que /tmp és un directori i no un fitxer, mentre que /usr/bin/java sí és un fitxer.

Per què isDirectory() i isFile() retornin cert, és necessari que el camí correspongui a un objecte existent al sistema de fitxers. És a dir, si ! fitxer.exists() llavors segur que ! fitxer.isFile() && ! fitxer.isDirectory().

Més coses amb File

El cert és que File ens ofereix unes quantes funcionalitats prou interessants.

Per exemple, considera el següent codi:

 1/* programa que demostra algunes funcionalitats interessants de File */
 2import java.io.File;
 3public class ExploraFile {
 4    public static void main(String[] args) {
 5        File fitxer = new File("ExploraFile.java");
 6        System.out.println("Camí relatiu: " + fitxer);
 7        System.out.println("Camí absolut: " + fitxer.getAbsolutePath());
 8        System.out.println("Nom         : " + fitxer.getName());
 9        System.out.println("Longitud    : " + fitxer.length());
10        File carpeta = fitxer.getAbsoluteFile().getParentFile();
11        System.out.println("Carpeta     : " + carpeta);
12        System.out.println("Continguts dins la carpeta " + carpeta);
13        String[] continguts = carpeta.list();
14        for (int i = 0; i < continguts.length; i++) {
15            String item = continguts[i];
16            System.out.println("\t" + item);
17        }
18    }
19}

Suposant que l'executem a la carpeta /tmp/carpeta/subcarpeta, ens trobem el següent:

$ ls
ExploraFile.class   ExploraFile.java
$ java ExploraFile
Camí relatiu: ExploraFile.java
Camí absolut: /tmp/carpeta/subcarpeta/ExploraFile.java
Nom         : ExploraFile.java
Longitud    : 803
Carpeta:    : /tmp/carpeta/subcarpeta
Continguts dins la carpeta /tmp/carpeta/subcarpeta
    ExploraFile.class
    ExploraFile.java

Pots trobar el detall d'aquests mòduls i encara més a la documentació oficial de File.

Una mica d'ordre

Com hem vist a l'exemple anterior, podem llistar el contingut d'una carpeta amb la funció File.list()

A l'exemple, hem volgut llistar la carpeta que contenia el codi del nostre programa, però podríem haver fet servir qualsevol altra carpeta.

En cas que el que ens interessi sigui la carpeta actual, podem aconseguir-la amb new File(".").

Així, un programa que llisti el contingut de la carpeta actual podria tenir el següent aspecte:

 1/* Programa que mostra els continguts de la carpeta actual */
 2import java.io.File;
 3public class Llista {
 4    public static void main(String[] args) {
 5        File carpeta = new File(".");
 6        String[] continguts = carpeta.list();
 7        for (int i = 0; i < continguts.length; i++) {
 8            String item = continguts[i];
 9            System.out.println("\t" + item);
10        }
11    }
12}

Ara observa aquesta execució:

$ touch un dos tres     # creem tres fitxers buits
$ ls
dos  Llista.class  Llista.java  tres  un
$ java Llista
    Llista.java
    dos
    un
    Llista.class
    tres

Molt bé, funciona, però… l'ordre en que apareixen els fitxers no és gaire… ordenat, oi? Com és possible que un surti abans de tres però després de dos?

 1/* Programa que mostra els continguts de la carpeta actual ordenats alfabèticament */
 2import java.io.File;
 3import java.util.Arrays;
 4public class Llista {
 5    public static void main(String[] args) {
 6        File carpeta = new File(".");
 7        String[] continguts = carpeta.list();
 8        Arrays.sort(continguts);
 9        for (int i = 0; i < continguts.length; i++) {
10            String item = continguts[i];
11            System.out.println("\t" + item);
12        }
13    }
14}

La biblioteca de Java ens ofereix, entre tantes altres coses, una manera senzilla d'ordenar el contingut d'un array. En aquest cas, ho fem amb el procediment Arrays.sort() que podem trobar a la documentació oficial

Ara, si executem el nostre programa, ens trobem la sortida ordenada alfabèticament.

$ java Llista
    Llista.class
    Llista.java
    dos
    tres
    un

No està malament, oi?

Un altre iterador

Fins ara no t'he presentat una modalitat de bucle que pot ser interessant en determinats contexts. Java l'anomena enhanced for o for millorat.

Veiem com funciona a partir de l'exemple anterior:

 1/* Programa que mostra els continguts de la carpeta actual ordenats alfabèticament */
 2import java.io.File;
 3import java.util.Arrays;
 4public class Llista {
 5    public static void main(String[] args) {
 6        File carpeta = new File(".");
 7        String[] continguts = carpeta.list();
 8        Arrays.sort(continguts);
 9        for (String item: continguts) {
10            System.out.println("\t" + item);
11        }
12    }
13}

A la versió anterior fèiem servir la variable de recorregut i només per a poder obtenir cada element del bucle. A banda d'això, no fèiem servir la i per a res. Quan ens passa això en Java, sovint podem fer servir aquesta nova modalitat de for.

El seu funcionament és molt simple. Declares una variable del tipus dels elements de l'array, afegeixes : i tot seguit l'array que vols recórrer. La variable declarada anirà recollint un valor de l'array de manera seqüencial, a cada iteració.

És còmode? Sí. Has de fer-ho servir obligatòriament? No.

Malauradament encara no funciona amb Strings i de moment ens tocarà haver de declarar la variable de recorregut per passar per cada caràcter d'un text.

Una nova biblioteca

Java ha incorporat una nova manera de gestionar fitxers a partir de la biblioteca nio (new I/O).

L'he deixada fora d'aquests apunts perquè algunes de les funcionalitats inclouen l'ús de streams i funcions lambda, característiques amb una complexitat una mica per sobre de la que aquesta introducció pot requerir.

En tot cas, si la curiositat et pot, considera consultar la documentació oficial de la biblioteca nio. Ja t'aviso que no estaràs perdent el temps.
